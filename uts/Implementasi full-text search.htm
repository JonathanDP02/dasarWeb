Bagus
Kode SQL yang kamu tulis itu adalah implementasi full-text search PostgreSQL dengan bahasa Indonesia pada tabel proposal. Mari kita bahas langkah demi langkah agar kamu benar-benar paham fungsinya

1. Menambahkan kolom search_vector

sql
ALTER TABLE proposal ADD COLUMN search_vector tsvector;

Ini menambahkan kolom baru bernama search_vector pada tabel proposal.
Kolom ini akan menyimpan hasil indeks teks (tokenized & stemmed) dari kolom seperti judul_proposal dan deskripsi.
Jenis datanya tsvector, khusus digunakan untuk pencarian teks di PostgreSQL.

2. Mengisi kolom search_vector

sql
UPDATE proposal
SET search_vector =
setweight(to_tsvector('indonesian', coalesce(judul_proposal,'')), 'A') ||
setweight(to_tsvector('indonesian', coalesce(deskripsi,'')), 'D');

Ini mengisi kolom search_vector dengan gabungan dua bagian:

  • judul_proposal diberi bobot A (tinggi)

  • deskripsi diberi bobot D (rendah)

Fungsi:

  • to_tsvector('indonesian', teks) mengubah teks menjadi token berdasarkan bahasa Indonesia (misalnya keamanan jaringan jadi token [keaman, jaring]).

  • setweight() memberi bobot (A, B, C, D; A paling penting).

  • || menggabungkan dua tsvector.

  • coalesce(teks,'') agar tidak error kalau kolomnya NULL.

Tujuannya: hasil pencarian akan lebih menekankan kecocokan di judul daripada di deskripsi.

3. Membuat index GIN

sql
CREATE INDEX idx_proposal_search
ON proposal
USING gin(search_vector);

Membuat index GIN (Generalized Inverted Index) pada kolom search_vector.
Index ini mempercepat pencarian teks penuh (full-text search) secara drastis.

4. Melakukan pencarian dengan ranking

sql
SELECT
proposal_id,
judul_proposal,
ts_rank(search_vector, to_tsquery('indonesian', 'jaringan & keamanan')) AS rank
FROM proposal
WHERE search_vector @@ to_tsquery('indonesian', 'jaringan & keamanan')
ORDER BY rank DESC;

Ini mencari proposal yang mengandung kedua kata jaringan dan keamanan (& artinya AND).
Fungsi:

  • @@ operator pencocokan teks (mirip LIKE, tapi untuk tsvector).

  • to_tsquery('indonesian','jaringan & keamanan') ubah query teks jadi format pencarian.

  • ts_rank() menghitung relevansi (semakin tinggi semakin cocok).

  • ORDER BY rank DESC urutkan berdasarkan tingkat kecocokan.

5. Contoh pencarian lainnya

a. AND (harus ada dua kata)

sql
SELECT judul_proposal FROM proposal
WHERE search_vector @@ to_tsquery('indonesian', 'sistem & rekomendasi');

Hanya menampilkan proposal yang mengandung kedua kata sistem dan rekomendasi.

b. OR (boleh salah satu)

sql
SELECT judul_proposal FROM proposal
WHERE search_vector @@ to_tsquery('indonesian', 'chatbot | rekomendasi');

Mencari proposal yang mengandung kata chatbot atau rekomendasi.

c. Prefix search (kata yang diawali dengan)

sql
SELECT judul_proposal FROM proposal
WHERE search_vector @@ to_tsquery('indonesian', 'rekomendasi:*');

Mencari semua kata yang diawali dengan rekomendasi, misalnya rekomendasi, rekomendasikan, dll.

6. Menampilkan hasil dengan snippet (highlight)

sql
SELECT
proposal_id,
judul_proposal,
ts_rank(search_vector, to_tsquery('indonesian', 'jaringan & keamanan')) AS rank,
ts_headline(
'indonesian',
deskripsi,
to_tsquery('indonesian','jaringan & keamanan'),
'StartSel=<b>,StopSel=</b>,MinWords=5,MaxWords=15,MaxFragments=2'

ORDER BY rank DESC
LIMIT 5;

Ini menampilkan hasil pencarian dengan cuplikan (highlight) kata yang cocok.

Fungsi ts_headline:

  • Membuat potongan teks (snippet) dari kolom deskripsi.

  • Menandai kata yang cocok dengan tag <b> dan </b>.

  • MinWords=5, MaxWords=15 mengatur panjang potongan.

  • MaxFragments=2 maksimal dua potongan per hasil.

2 Mengapa indeks GIN penting untuk Full-Text Search?

Karena tanpa indeks GIN, PostgreSQL tetap bisa melakukan pencarian FTS (@@), tapi harus melakukan sequential scan terhadap semua baris di tabel.

GIN (Generalized Inverted Index) bekerja seperti kamus balik:

  • Menyimpan setiap token (kata) dari tsvector beserta daftar dokumen (row) yang mengandungnya.

  • Jadi saat kamu mencari kata jaringan, PostgreSQL langsung tahu baris mana saja yang mengandung token itu.

Keuntungannya:

  • Pencarian ribuan teks bisa jadi milidetik.

  • Skalabilitas tinggi untuk dataset besar.

  • Efisien untuk operasi @@ dan @? (query FTS).

Oke, saya jawab satu per satu sesuai permintaan langsung, jelas, dan singkat.

1) Keunggulan to_tsvector() dibanding LIKE '%...%'

  • Performa: to_tsvector() + index (GIN) memungkinkan index scan; LIKE '%...%' biasanya memaksa full table scan jauh lebih lambat untuk tabel besar.

  • Stemming / linguistik: to_tsvector() melakukan stemming/normalisasi (mis. direkomendasikan, rekomendasi dianggap serupa). LIKE mencari literal substring tidak peka bentuk kata.

  • Operator logika & fleksibilitas: FTS mendukung & (AND), | (OR), ! (NOT), prefix :* LIKE hanya wildcard/substring.

  • Ranking: Dengan ts_rank() kamu bisa memberi skor relevansi dan urutkan hasil; LIKE tidak punya mekanisme relevansi.

  • Highlight/snippet: ts_headline() bisa menghasilkan potongan teks dengan highlight hasil pencarian; LIKE tidak menyediakan itu.

  • Kesimpulannya: Untuk pencarian teks non-trivial dan dataset mediumbesar, to_tsvector() jauh lebih benar dan efisien daripada LIKE '%...%'.

2) Mengapa indeks GIN penting untuk Full-Text Search

  • Cara kerja GIN: menyimpan inverted index peta dari token kata daftar baris yang mengandung token itu. Saat mencari, DB langsung lihat daftar baris terkait token tanpa memindai seluruh tabel.

  • Performa: tanpa GIN, operator @@ harus sequential-scan seluruh tabel lambat. Dengan GIN, pencarian jadi cepat (skala besar tetap responsif).

  • Kesesuaian dengan tsvector: GIN dirancang untuk struktur seperti tsvector yang berisi banyak token per baris; efisien menyimpan dan mencari token-token tersebut.

  • Kompromi: GIN meningkatkan kecepatan baca (search) tapi menambah overhead saat INSERT/UPDATE (index harus di-update). Umumnya trade-off yang diterima untuk aplikasi baca-intensif.

  • Kesimpulan: Jika mau FTS yang cepat dan skala, buat index GIN pada kolom tsvector itu nyaris wajib untuk produksi.

Berikut jawaban singkat dan jelas

1 Kelebihan menggunakan JSONB dibanding banyak kolom tabel

  • Fleksibel bisa simpan data dinamis tanpa ubah struktur tabel.

  • Mendukung nested data cocok untuk data kompleks (array, objek).

  • Lebih ringkas tidak perlu banyak kolom untuk atribut opsional.

  • Bisa di-query langsung dengan operator JSON (->, ->>, @>).

  • Indeks GIN bisa dibuat agar pencarian tetap cepat.

2 Perbedaan antara operator -> dan ->>

OperatorHasilTipe hasil
->Mengambil nilai JSON (tetap bertipe JSON)JSON / JSONB
->>Mengambil nilai teks dari JSONtext

Contoh:

sql
data -> 'nama'   -- hasilnya: "Jonathan"
data ->> 'nama'  -- hasilnya: Jonathan

3 Cara mempercepat pencarian data JSONB di kolom besar

  • Buat indeks GIN pada kolom JSONB:

    sql
    CREATE INDEX idx_data_jsonb ON tabel USING gin(data);

  • Gunakan operator yang mendukung indeks, seperti @>, ?, ?|, ?&.

  • Hindari fungsi yang memaksa full-scan (jsonb_to_record, jsonb_each) pada data besar.

  • Bisa juga buat partial index untuk elemen JSON tertentu jika sering di-query.
